# 从 YesPlayMusic 到 CloudSound：一次前端功能扩展与双端适配实战

## 背景

这次工作的基础项目是 [qier222/YesPlayMusic](https://github.com/qier222/YesPlayMusic)。

它本身已经是一个成熟的 Vue 2 音乐播放器项目，拥有完整的页面结构、播放器能力、歌词页、歌单页、搜索页，以及 Electron 和 Web 两种运行形态。

这次我的目标不是从零做播放器，而是在一个成熟项目上做一轮真正可落地的二次开发，重点包括：

1. 跑通本地开发环境
2. 理解项目结构
3. 增加更贴近日常使用的功能
4. 对 Web 端做移动端和桌面端双端适配

## 一、先把项目跑起来

项目一开始并不能直接顺利运行，主要问题集中在环境层：

- Node.js 版本需要匹配老项目依赖
- 依赖安装后需要兼容当前本机环境
- 本地 Web 服务和本地 API 服务需要明确分开启动

### 为什么跑不起来

老项目通常会有几个常见问题：

- 依赖树和当前 Node 版本不兼容
- 前端开发服务能起，但接口代理不通
- 浏览器打开后实际上是旧缓存或编译失败页面

### 我做了什么

- 调整本地 Node 运行版本
- 安装项目依赖
- 让本地 Web 服务跑在固定端口
- 确认本地 API 和 Web 前端的职责划分
- 输出了一份单独的本地部署记录文档

对应文档：

- [LOCAL_SETUP_REPORT.md](../LOCAL_SETUP_REPORT.md)

## 二、理解项目结构

这个项目本质上是一个：

- 前端 Web 应用
- 本地/远程 API 配合
- 第三方音乐接口代理
- 本地缓存与状态存储

的综合项目。

从代码结构上，比较关键的几个点是：

- `src/App.vue`
  负责全局框架，包括顶部导航、主内容区、底部播放器、歌词页浮层
- `src/components/Player.vue`
  负责底部播放器交互
- `src/components/Navbar.vue`
  负责顶部导航和搜索
- `src/views/*`
  是各个页面，比如首页、发现、资料库、专辑、歌单、艺人、设置等
- `src/utils/Player.js`
  是播放器真正的播放逻辑核心
- `src/utils/db.js`
  是本地 Dexie 数据库能力

## 三、功能扩展：缓存管理增强

第一个真正落地的功能，是设置页缓存管理增强。

原来的缓存区域偏简单，也比较突兀。我这次做了两个方向：

### 1. 功能增强

- 增加缓存总览
- 增加各缓存分类统计
- 增加单项清理
- 增加刷新统计
- 避免“清缓存”误伤本地播放历史

### 2. 视觉统一

- 去掉灰色大底板
- 降低字体粗细
- 与歌词等其他设置 section 保持同一视觉语言

关键文件：

- `src/views/settings.vue`
- `src/utils/db.js`

## 四、功能扩展：本地播放历史与听歌统计

第二个扩展方向，是把“本地播放记录”这件事做起来。

很多 Web 音乐播放器只显示云端历史，但本地使用时其实更想知道：

- 最近听了什么
- 听了多久
- 哪些歌重复播放最多

所以这次我做了本地统计方案。

### 核心实现思路

在播放器播放过程中记录：

- 歌曲 ID
- 已播放秒数
- 是否完整播放
- 最近播放时间

并且设置了一个“有效播放阈值”，避免误触播放几秒就被记成一次完整播放。

### 新增能力

- 本地播放历史摘要
- 总播放次数
- 总听歌时长
- 最近播放歌曲
- 本地历史与云端历史切换
- 清空本地历史

关键文件：

- `src/utils/Player.js`
- `src/utils/db.js`
- `src/store/actions.js`
- `src/store/state.js`
- `src/views/library.vue`

## 五、功能扩展：均衡器和音质入口内嵌到播放器

你后来提了一个非常明确的交互方向：

不要把均衡器和音质入口放在设置页里，
而是像很多真正的音乐播放器一样，直接内嵌到播放器本体里。

这个方向我很认同，因为它更符合使用场景。

### 我做了什么

- 在播放器右侧增加音质按钮
- 增加均衡器按钮
- 点击后展开音频面板
- 面板内包含：
  - 音质切换
  - 均衡器开关
  - 均衡器预设
  - 10 段手动调节
  - 重置入口

### 更关键的是

这次不是只做 UI，我把 Web Audio 的均衡器链路也接进去了。

也就是说：

- 设置会进本地状态
- 播放器会把这些设置真正应用到当前音频节点

关键文件：

- `src/components/Player.vue`
- `src/utils/Player.js`
- `src/store/initLocalStorage.js`

## 六、最花时间的一步：移动端和桌面端双端适配

后面最大的工作量，其实来自响应式改造。

因为这个项目原本明显更偏桌面端：

- 顶部导航布局默认按 PC 宽度设计
- 主内容留白按桌面计算
- 底部播放器高度和按钮密度按桌面设计
- 专辑、歌单、艺人详情页都有比较重的横向信息结构

而当你拿手机局域网访问时，问题就会一下子暴露出来：

- 顶部导航挤成一团
- 播放栏挡住内容
- 详情页横向溢出
- 页面结构看起来像“桌面缩小版”，而不是“真正的移动端页面”

### 这一轮改造的核心原则

不是把 PC 页面硬缩小，
而是按“不同终端的阅读和触控方式”重新组织布局。

### 我做了哪些结构级调整

#### 1. 全局框架

- 用 CSS 变量统一控制导航高度和播放器占位高度
- 让 `main` 的 padding 根据终端自动变化
- 给移动端留出安全区

文件：

- `src/App.vue`

#### 2. 顶部导航

- 手机和平板改成更清晰的两层结构
- 上层放返回、前进、搜索
- 下层放导航 Tab
- 搜索框在小屏下做成更轻量的展开式交互

文件：

- `src/components/Navbar.vue`

#### 3. 底部播放器

- 移动端改成悬浮 mini-player
- 抬高位置，避开手机浏览器底部栏
- 小屏下隐藏一部分次级按钮
- 保留核心播放控制和新的音频面板入口

文件：

- `src/components/Player.vue`

#### 4. 通用卡片与列表

- 卡片在移动端自动减少列数
- 播放按钮适配触屏
- TrackList 在小屏自动从多列切到单列
- TrackListItem 压缩信息密度，隐藏非核心列

文件：

- `src/components/Cover.vue`
- `src/components/CoverRow.vue`
- `src/components/TrackList.vue`
- `src/components/TrackListItem.vue`

#### 5. 核心页面响应式

这次优先处理了：

- 首页 `home`
- 发现页 `explore`
- 资料库 `library`
- 专辑页 `album`
- 歌单页 `playlist`
- 艺人页 `artist`
- 搜索页 `search`

这些页面现在至少已经从“桌面压缩版”进入“可用的移动端页面”阶段。

## 七、这次开发里比较重要的几个技术点

### 1. 在老 Vue 2 项目里做增量式响应式改造

这种项目最难的不是写一套新页面，
而是在不推翻原逻辑的前提下做分层适配。

关键是：

- 先动全局骨架
- 再动共用组件
- 最后补具体页面

这样不会陷入每个页面各写一套“特判样式”的泥潭。

### 2. 把“缓存”和“本地统计”分开

这是一个产品上很容易混淆，但技术上必须明确边界的地方。

如果用户点“清缓存”，通常希望清的是：

- 歌曲缓存
- 歌词缓存
- 详情缓存

而不是：

- 本地听歌记录
- 本地统计数据

所以我在本地数据库层做了明确拆分。

### 3. 均衡器不是 UI，而是播放链路能力

如果只做前端面板，那只是“看起来有均衡器”。

真正有价值的是：

- 播放器拿到设置
- Howler / Web Audio 节点建立滤波链
- 播放过程中真正生效

这一点是这次功能里最像“播放器产品能力”的部分。

## 八、遇到的问题

### 1. 局域网手机访问看到旧编译错误

一开始手机里看到的是 `settings.vue` 的编译报错页，但那其实并不是当前代码状态，而是开发服务热更新阶段留下的旧错误页面。

解决方式：

- 先确保本地代码已经修复
- 重新通过构建校验
- 手机端强刷页面或重新打开地址

### 2. 响应式不能只改一个页面

比如你看到的是首页导航和专辑页的问题，
但真正动手时发现根源在：

- `App.vue`
- `Navbar.vue`
- `Player.vue`
- 通用卡片/列表组件

所以必须按组件树自上而下改。

### 3. Vue + ESLint + Prettier 联合校验

每轮改完布局后都要过一次构建，
因为：

- Vue 模板有属性顺序规则
- Prettier 对换行非常严格
- 老项目配置一旦不满足就直接编译失败

这次也确实多次卡在格式校验上，但这种校验能保证最终仓库状态干净。
