# CloudSound 本地启动与环境整理记录

## 最终可用结果

- 前端开发地址：`http://localhost:20202/`
- 本地 API 地址：`http://localhost:3000/`
- 当前项目可以完成：
  - `yarn serve`
  - `yarn build`
  - 局域网访问前端页面

## 一开始为什么跑不起来

这次本地启动问题不是单点故障，而是多个环境因素叠加：

### 1. Node.js 版本不匹配

项目 `package.json` 要求：

```json
"engines": {
  "node": "14 || 16"
}
```

而本机初始环境不是这一范围内的稳定版本。对于这个基于 `Vue CLI 4 + Vue 2 + Electron 13` 的老项目来说，Node 版本过新时很容易出现依赖兼容问题。

### 2. 依赖没有完整落地

初始状态下缺少：

- `.env`
- `node_modules`

项目虽然已经 clone 下来，但并不具备直接启动条件。

### 3. Electron 依赖链会影响 Web 端运行

虽然这次主要跑的是 Web 端，但仓库本身集成了 `vue-cli-plugin-electron-builder`。依赖安装不完整时，Electron 相关校验同样会影响 `yarn serve` 的执行。

### 4. 默认端口在当前机器上不稳定

原项目示例里常见的是 `20201`，但在这台机器环境里，改到 `20202` 后前端开发服务更稳定。

### 5. 本地 API 与前端是两套进程

这个项目不是一个纯静态页面，它依赖本地或远程 API。因此即使前端起了，也要区分：

- 页面服务有没有启动成功
- API 服务有没有启动成功
- API 能不能正常访问第三方上游服务

## 我做了哪些处理

### 1. 安装并整理项目依赖

完成了依赖安装，让项目具备：

- 前端开发启动能力
- 生产构建能力
- 后续功能开发能力

### 2. 调整本地开发端口

为了提高本地稳定性，前端开发端口最终使用：

```env
DEV_SERVER_PORT=20202
```

### 3. 区分前端与本地 API 的运行职责

当前运行结构可以理解为：

- `localhost:20202`：前端 Web 应用
- `localhost:3000`：本地 API 服务

前端负责界面和交互，本地 API 负责代理第三方接口与数据请求。

### 4. 反复验证构建与运行

后续每轮功能改动后，我都通过 `yarn build` 做了完整校验，确保：

- Vue 模板通过编译
- ESLint / Prettier 规则满足
- 样式改造没有引入新的构建错误

## 当前项目如何启动

### 启动前端

```bash
yarn serve
```

### 启动生产构建

```bash
yarn build
```

### 启动本地 API

```bash
npm run netease_api:run
```

或者：

```bash
yarn netease_api:run
```

## 后续如果再次启动失败，优先排查什么

### 1. 先看 Node 版本

优先确保使用的是：

- Node 14
- 或 Node 16

### 2. 再看依赖是否完整

重点检查：

- `node_modules` 是否完整
- 是否误删锁文件
- 是否发生依赖安装中断

### 3. 再看端口占用

重点确认：

- `20202` 是否被其他进程占用
- `3000` 是否已被其他服务占用

### 4. 最后再看网络问题

如果页面能打开，但接口数据异常，就要区分：

- 是本地 API 没起
- 还是本地 API 访问上游第三方服务受限
